## 一、 前言与约定

本说明书详细解释由 cake1 创建的一套标准化任务触发模板。该模板旨在帮助地图制作者在自定义单人任务地图中，高效、逻辑清晰地实现多个连续任务目标的触发、管理与播报。

在阅读本文档前，请先了解以下约定：

- **obX** 表示第 X 个任务目标，例如 `ob1` 表示任务一，`ob2` 表示任务二，以此类推。
- 所有带下划线 `_` 的触发器（如 `[ObX]_Normal Action`、`[ObX]_Failed because 0`）在模板中**默认处于未启用状态**，供使用者根据需要自行启用以丰富任务细节。

## 二、 核心组件详解

模板由这些核心部分构成：局部变量（Variable）、触发器（Trigger）。

### 1. 局部变量

局部变量是整个模板的逻辑基石，用于记录任务的当前状态、控制流程冷却以及宣告最终胜负。所有变量均在 `任务目标模板.ini` 文件的 `[VariableNames]` 节中定义。**使用者应严格遵循下表中的说明进行操作，除非完全理解其连锁反应，否则不应随意修改。**

### 2. 冷却机制

模板包含两个重要的全局冷却机制，由 `[MissionTools]` 类触发器管理：

- **任务循环延迟 (`Objective Repeat Delay`)**：对应触发器 `MT00003`。其条件为局部变量 `[MissionTools] Objective Repeat` 为 `0`，且流逝时间达到 `300`。满足条件后，执行动作将该局部变量置 `1`。每个任务播报触发时都会等待此变量为 `1`。
- **任务介绍延迟 (`Intro Repeat Delay`)**：对应触发器 `MT00004`。其条件为局部变量 `[MissionTools] Intro Delay` 为 `0`，且流逝时间达到 `20`。满足条件后，执行动作将该局部变量置 `1`。每个过场触发时都会等待此变量为 `1`。

### 3. 触发器命名规范与结构

模板中的触发器遵循严格的命名规则：

- **`MT`** 前缀：代表 **MissionTools**，即全局工具触发器，用于控制冷却时间、宣告胜负、调试等。
- **`OBX`** 前缀：代表 **Objective X**，即与特定任务相关的触发器。例如 `OB1` 对应任务一，`OB2` 对应任务二。
- **`OBX` 过场序列**：以 `[ObX][Intro]0.` 开始，以 `[ObX][Intro]z.` 结束。`0.` 表示第一个过场事件，`z.` 表示最后一个。使用者可在两者之间插入任意数量的自定义触发器（如 `[Ob1][Intro]1.`、`[Ob1][Intro]2.` 等）来丰富过场动画，并通过触发结果（`53 - 允许触发`）将它们串联起来。

## 三、 标准任务流程详解 (以 ob1 为例)

一个任务从启动到完成的完整生命周期如下：

1. **启动任务开始过场 (`Start Intro ObX`)**：

   - **触发器**：`[MissionTools]Start Intro Ob1` (`OB10001`)
   - **条件**：对于第一个任务，其条件是 `8 - 任何事件`，即游戏开始时自动触发。对于后续任务（如 `OB20001`），其条件为前一个任务的 `Completed` 局部变量被设置为 `1`。
   - **动作**：执行 `53 - 允许触发`，启动 `[Ob1][Intro]0.` 触发器。
2. **播放任务开始过场 (`[ObX][Intro]0.` ~ `z.`)**：

   - **触发器**：一系列连续的触发器，例如 `OB10002` (`[Ob1][Intro]0.`) 和 `OB10003` (`[Ob1][Intro]z.`)。
   - **条件**：每个此类触发器都包含两个条件：`36 - 局部变量被设置`（对应 `[MissionTools] Intro Delay` 变量为 `1`）和 `13 - 流逝时间`（例如 `20`），以确保过场之间有足够的间隔，避免重叠。
   - **动作**：执行过场所需的动作，如显示文本、播放语音、移动镜头等。动作执行后，会清除 `[MissionTools] Intro Delay` 变量以开始新一轮冷却，并执行 `53 - 允许触发` 来启动序列中的下一个触发器。
3. **任务正式激活 (`Available`)**：

   - **触发器**：`[ObX][Intro]z.` (`OB10003`)
   - **动作**：在序列的最后一个过场触发器中，除了启动后续流程外，还会执行 `56 - 设置局部变量`，将 `[Mission] Objective X Available` 设置为 `1`。此时，任务目标正式向玩家开放，相关的失败条件和常规事件开始生效。
4. **任务完成 (`Complete`)**：

   - **触发器**：`[ObX]Complete` (`OB10004`)
   - **条件**：默认为 `48 - 被任何事物摧毁`。使用者可根据实际任务需求修改此条件。
   - **动作**：执行 `56 - 设置局部变量`，将 `[Mission] Objective X Completed` 设置为 `1`，并显示任务完成信息（`11 - 文本触发事件` 和 `21 - 播放语音`）。
   - **⚠️ 重要注意事项（关于"抢跑"问题）**：
     > 默认的完成条件 `48 - 被任何事物摧毁` 在顺序执行任务时没有问题。但是，如果玩家在任务一阶段就提前摧毁了任务二要求的目标建筑，那么当任务二真正激活时，由于该建筑已不存在，`48` 事件将无法再次触发，从而导致卡关。
     > **解决方案**：如果你的地图存在玩家"抢跑"的可能，请新建一个触发器，当该目标建筑被摧毁时，设置一个自定义的局部变量（例如 `BuildingADestroyed`）。然后，将 `[ObX]Complete` 触发器的条件从 `48 - 被任何事物摧毁` 改为 `36 - 局部变量被设置`，并指向你新建的局部变量。这样，无论建筑在何时被摧毁，其状态都会被正确记录，任务二到来时即可顺利判定为完成。
     >
5. **任务失败 (`Failed`)**：

   - **触发器**：`[ObX]_Failed because 0` (`OB10005`)
   - **条件**：此触发器默认未启用。其预设条件由三个部分组成，需同时满足：① `[Mission] Objective X Available` 为 `1`（任务已激活）；② `[Mission] Objective X Completed` 为 `0`（任务未完成）；③ 关联对象被摧毁（`48 - 被任何事物摧毁`）。这三个条件确保了失败逻辑仅在任务进行中且目标被摧毁时触发。
   - **动作**：播放警告音效（`19 - 播放音效`），显示失败原因文本（`11 - 文本触发事件`），并将 `[Action] Mission_Failed` 局部变量设置为 `1`（`56 - 设置局部变量`）。
6. **任务进行中的常规事件 (`Normal Action`)**：

   - **触发器**：`[ObX]_Normal Action` (`OB10006`)
   - **条件**：默认未启用。其预设条件为：`[Mission] Objective X Available` 为 `1`，且流逝时间大于 `0`。这意味着一旦任务激活，此触发就会开始满足条件。
   - **用途示例**：
     > 你可以启用此触发器并修改其动作，以实现仅在**任务X进行期间**才会发生的逻辑。例如，在任务二开始后，你希望玩家在港口得到一波援军，就可以在 `ob2` 的 `Normal Action` 触发器里写上对应的刷兵动作。如果你希望此效果仅在任务二期间生效，需要增加一个条件：`[Mission] Objective 2 Completed` 局部变量必须为 `0`。
     >
7. **任务胜利宣告 (`Mission Accomplished`)**：

   - **触发器**：`[MissionTools]Action_MissionAccomplished 0` (`MT00005`)
   - **条件**：检测到 `[Action] Mission_Accomplished` 局部变量被设置为 `1`。
   - **动作**：允许后续的胜利处理触发器（`53 - 允许触发`），如播放欢呼动作（`MT00006`）和胜利语音（`MT00007`）。
   - **如何使用**：当你的任务流程需要宣布玩家胜利时（例如所有任务均已完成），只需使用触发动作 `56 - 设置局部变量` 将 `[Action] Mission_Accomplished` 设置为 `1` 即可。
8. **任务失败宣告 (`Mission Failed`)**：

   - **触发器**：`[MissionTools]Action_MissionFailed` (`MT00008`)
   - **条件**：检测到 `[Action] Mission_Failed` 局部变量被设置为 `1`。
   - **动作**：执行 `1 - 胜利者是...`（参数为玩家所属方以外的其他方，例如 `12`），并播放失败语音。

## 四、 调试模式 (Debug Mode)

模板内置了简易的调试功能，方便地图作者在测试时快速验证。

- **激活方式**：触发器 `[Debug]Set Debug Mode` (`MT00002`) 默认条件是玩家选择了"古巴"国家。当此条件满足时，执行动作 `56 - 设置局部变量`，将 `[Mission] Debug Mode`（局部变量 `0`）设置为 `1`。
- **自定义激活条件**：你可以修改 `[Debug]Set Debug Mode` 触发器的条件，将其改为一个在你的地图中玩家**不会**选择到的国家。这样，当你自己测试时，只需选择这个特定国家，即可进入调试模式。
- **调试动作示例**：模板中包含一个 `[Debug]Debug View` 触发 (`MT00001`)。当 `[Mission] Debug Mode` 为 `1` 时，该触发会执行 `16 - 显示全地图`，为你清除战争迷雾，方便观察全局。

## 五、 自定义与扩展指南

### 1. 修改任务数量

- **减少任务数量**：如果你只想编写三个任务，只需要**禁用**第四个任务的启动触发器 `[MissionTools]Start Intro Ob4` (`OB40001`) 即可。任务流程将在任务三完成后自然中断。
- **增加过场内容**：你可以在 `0.` 和 `z.` 之间插入更多触发器来丰富过场。例如：

  ```
  [ObX][Intro]0.
  [ObX][Intro]1.（过场：谭雅出现）
  [ObX][Intro]2.（过场：谭雅进入五角大楼）
  [ObX][Intro]3.（文字信息：我们必须保卫五角大楼，保证谭雅的安全）
  [ObX][Intro]z.
  ```

  请确保每个前置触发器都通过 `53 - 允许触发` 动作来启动下一个触发器，将它们链接起来。

### 2. 修改文本信息

所有在游戏中显示的文本，均在 `任务目标模板csf.ini` 文件中定义。请使用 CSF 编辑器打开并修改对应的条目。各条目含义如下：

| CSF 标签                | 用途说明                               | 来源                    |
| :---------------------- | :------------------------------------- | :---------------------- |
| `ob1In0`              | 任务一介绍：开始第一个任务目标时显示   | `任务目标模板csf.ini` |
| `ob1Complete`         | 任务目标一完成时显示的消息             | `任务目标模板csf.ini` |
| `Ob1failed0`          | 任务目标一失败时显示的原因             | `任务目标模板csf.ini` |
| `ob1`                 | 任务目标一的具体内容文本               | `任务目标模板csf.ini` |
| `ob2In0` ~ `ob5`    | 任务二至任务五的相关消息，命名规则同上 | `任务目标模板csf.ini` |
| `MissionAccomplished` | 任务完成！你成功完成了所有任务目标！   | `任务目标模板csf.ini` |

### 3. 实现并行任务（无顺序要求）

如果你希望任务二和任务三可以并行进行（即玩家完成任务的顺序不受限制，任务四则必须等待两者都完成后才开放），需要进行以下修改：

1. **修改任务三的启动条件**：找到触发器 `[MissionTools]Start Intro Ob3` (`OB30001`)，将其条件从"局部变量 `[Mission] Objective 2 Completed` 被设置"（对应变量ID `4`）改为"局部变量 `[Mission] Objective 1 Completed` 被设置"（对应变量ID `2`）。这样，任务一完成后，任务二和任务三的启动过场将同时被允许。
2. **修改任务四的启动条件**：相应地，你需要将任务四的启动条件改为"局部变量 `[Mission] Objective 2 Completed` **和** `[Mission] Objective 3 Completed` **均**被设置"，以确保前两个任务都完成后再开放任务四。

### 4. 自定义过场延迟时间

模板中所有过场（Intro）的延迟均由 `[MissionTools]Intro Repeat Delay` 触发 (`MT00004`) 统一控制。如果你想为不同的过场设置不同的延迟，可以删除相关 `Intro` 触发器中关于 `[MissionTools] Intro Delay` 局部变量的条件和动作，直接使用 `13 - 流逝时间` 作为唯一的条件来控制间隔。

## 六、 结语

本模板将复杂的多任务流程抽象为可复用的模块，极大地简化了单人任务地图的制作难度。使用者通过理解和修改模板中的变量、触发器条件与动作，即可快速构建出制式任务目标系统。如遇到任何未在本文档中说明的疑问，请仔细查阅本说明书，或联系作者获取支持。
